Boltr — prediction prerequisites

Paths are on this machine only. Blank fields use defaults.

1. Model cache

Where boltr download / checkpoints live (boltz2_conf.safetensors, ccd.pkl, mols.tar, …). Same as boltr predict --cache-dir.

Blank → BOLTZ_CACHE, else ~/.cache/boltr.

LibTorch probe: set BOLTR to your tch-enabled boltr binary (e.g. …/target/release/boltr) so status can run boltr doctor --json, or install boltr on PATH.

Boltz model bootstrap (Boltr_Boltz_bootstrap): run ./Boltr_Boltz_bootstrap (or legacy ./Boltr_go) from the repo root to download Boltz2 weights into the cache, export safetensors, optionally verify keys and run tests, then write boltr_go_bootstrap.json. This is not the Python boltz preprocessor for templates/constraints; install that CLI separately for --preprocess boltz. Refresh status below to see the bootstrap record.

/home/YOURUSER/.cache/boltr

RunPod GPU — optional remote CUDA target over SSH. Configure the server with BOLTR_RUNPOD_HOST, optional BOLTR_RUNPOD_USER/PORT/KEY, plus BOLTR_RUNPOD_WORKDIR, BOLTR_RUNPOD_BOLTR, and BOLTR_RUNPOD_CACHE.

The browser never sends SSH credentials; it only asks this web server to probe or run the configured pod.

2. YAML + job folder

Job folder (optional): absolute path to the directory that already has your YAML, MSAs, templates, and optionally manifest.json + .npz. Lets the server check relative paths.

Blank → upload only; paths like ./msa.a3m won’t resolve.

/home/YOURUSER/Boltr/my-case

Pick the same YAML file below; with a job folder set, validation reads that path on disk.

Assume MSA server — relax missing local msa: files (still blocks missing templates, bad manifest, cache).

Preprocess bundle: run boltr preprocess --mode native or --mode boltz on the same machine so manifest.json + .npz sit next to your YAML, then point the job folder here. See docs/PREPROCESS_NATIVE.md in the repo.

3. Run predict

Runs boltr predict on this machine (needs BOLTR or boltr on PATH). Uses the same cache and job folder as above. Upload a YAML or set a job folder (uses input.yaml in that folder if no file is chosen).

Preprocess — if manifest.json is missing next to the YAML, choose auto, high-fidelity, native, or boltz (same as boltr predict --preprocess …). High-Fidelity Preprocess forces upstream Boltz and avoids native placeholder geometry. Native is protein-only: no non-empty templates:/constraints:, no ligands/DNA/RNA, no affinity bundle — use high-fidelity/boltz or place a Boltz-generated bundle next to the YAML for complexes. Boltz / auto need the upstream boltz CLI on PATH unless you set Bolt command to the full path (e.g. a venv …/bin/boltz). auto tries native first, then Boltz; it errors if neither applies (no silent “MSA-only” run).

Output format — structures are written as .cif (mmCIF) or .pdb per boltr predict --output-format. Only one run at a time. Logs stream below; on success, download a .tar.gz of the output folder. Disable with BOLTR_WEB_ENABLE_PREDICT=0.

Auto sets --device auto: LibTorch uses CUDA when available (unless free VRAM is below BOLTR_AUTO_MIN_FREE_VRAM_MB), otherwise CPU. With a single visible GPU, Boltz preprocess defaults to CPU so VRAM does not stack with LibTorch; use --preprocess-auto-boltz-gpu or BOLTR_AUTO_BOLTZ_GPU=1 to run Boltz on GPU. GPU requires CUDA. For a specific card index, use the CLI --device cuda:N or set BOLTR_DEVICE.
